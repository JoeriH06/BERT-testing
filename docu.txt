# 4. Construct Silver Text

## Objective
The objective of Step 4 is to transform structured project data (ID, title, content) into a single coherent document-level text string that can be processed consistently by downstream NLP components.

NLP models operate on text sequences, not tabular structures. Therefore, each project must be represented as one unified text document.

---

## Document Boundary Definition
In this system:

- One uploaded project = one document  
- One row in the bronze dataset = one document  
- The document boundary is fixed at the row level  

Fixing the boundary ensures processing is consistent and reproducible across all projects.

---

## Silver Schema
The Silver layer preserves raw fields for traceability while introducing a canonical text representation.

Each Silver record contains:

- `id` (metadata, not part of text processing)
- `raw_title`
- `raw_content`
- `silver_text`

The ID is kept for lineage and debugging, but it is not included inside the text string. IDs are not linguistic content and would introduce artificial tokens into downstream NLP steps.

---

## Why We Create One Coherent String
NLP pipelines expect a continuous sequence of text as input. Splitting title and content into separate processing streams introduces unnecessary complexity and ambiguity.

Using one canonical `silver_text` field:

- Ensures consistent downstream processing
- Avoids field-specific logic later in the pipeline
- Preserves document semantics
- Prevents non-linguistic metadata (e.g., IDs) from polluting the text representation

---

## Silver Text Construction Rules
The Silver text is constructed as:

Title  
(blank line)  
Content  

Implementation principles:

- Preserve original casing
- Preserve punctuation
- Preserve numbers
- Preserve paragraph structure
- Do not remove symbols or alter semantics
- Handle missing fields safely

If both title and content exist:

- `silver_text = title + "\n\n" + content`

If only one field exists, use that field as the document text.

# 5. Inspect and Normalize Text (Pre-NLP)

## Objective

The purpose of this step is to systematically inspect the constructed `silver_text` and apply justified, meaning-preserving normalization before any NLP techniques are used.

This step ensures:
- Reproducibility
- Transparency of preprocessing decisions
- Data quality control
- Protection of semantic content

Inspection always precedes cleaning.

---

## Why Inspection Is Necessary

Text extracted from PDFs or content management systems may contain structural and formatting artifacts such as:

- Empty or near-empty documents
- Extremely short or unusually long documents
- Excessive whitespace
- Repeated paragraphs
- Repeated headers or footers
- Broken formatting (e.g., one word per line)
- Encoding inconsistencies

Applying cleaning without inspection risks:
- Removing meaningful content
- Introducing bias
- Creating non-reproducible transformations

Therefore, inspection defines what normalization is justified.

---

## 5.1 Text Profiling

To make inspection reproducible, we introduce measurable metrics:

- Character length
- Word length
- Empty document detection
- Very short document detection
